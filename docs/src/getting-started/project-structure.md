# Project Structure

A Forge project is a single directory containing a `forge.toml` manifest and an `app/` directory. Everything outside `app/` and `public/` is configuration.

```
my-app/
├── forge.toml          # Project manifest and dependency declarations
├── foundry.lock        # Locked dependency tree (commit to version control)
├── app/
│   ├── routes.fx       # Route tree — the single source of truth for URLs
│   ├── schema.fx       # Database schema definitions
│   ├── auth.fx         # Auth instance configuration
│   ├── email.fx        # Email adapter configuration (optional)
│   ├── storage.fx      # Storage adapter configuration (optional)
│   ├── channels.fx     # Realtime channel definitions (optional)
│   ├── pages/          # Page components — one per route
│   ├── components/     # Shared UI components
│   ├── server/         # Server functions
│   ├── emails/         # Email template components
│   └── jobs/           # Background job definitions
└── public/             # Static assets served verbatim
```

## forge.toml

The project manifest. Declares the app name, version, deployment target, and Foundry dependencies.

```toml
[app]
name = "my-app"
version = "0.1.0"
target = "server"

[dependencies]
"forge:router" = "0.1.0-pre-alpha"
"forge:data" = "0.1.0-pre-alpha"
"forge:auth" = "0.1.0-pre-alpha"
```

## foundry.lock

The Foundry lockfile pins every dependency to its exact content hash. This ensures reproducible builds. Commit it to version control. Do not edit it by hand — `forge install` manages it.

## app/routes.fx

The single source of truth for all URL routes. Every URL your application responds to must be declared here. This is intentional: file-system routing obscures the URL structure; explicit routing makes it auditable.

## app/schema.fx

Database table definitions using `forge:data`'s `Schema` DSL. Running `forge migrate` reads this file and generates SQL migration files in `migrations/`.

## app/auth.fx

The configured `Auth` instance from `forge:auth`. Imported by server functions and middleware. Not bundled into client code.

## app/pages/

Page components, one per route. These are `.fx` components that export a default component. They may import from `app/components/` and call server functions.

## app/server/

Server function modules. Functions defined here with the `server` keyword are exposed as RPC endpoints. Client code that imports them receives a generated stub.

## app/emails/

Email template components. These use `"use module email"` and compile to HTML rather than browser DOM.

## app/jobs/

Background job definitions using `forge:jobs`'s `Job.define()`. Imported by server functions to enqueue work.

## public/

Static files served without processing. Images, fonts, `robots.txt`, `favicon.ico`. Files here are hashed and cache-busted in production builds.

## migrations/

Generated by `forge migrate`. SQL migration files that track the evolution of your database schema. Commit these — they are the audit trail for schema changes.
