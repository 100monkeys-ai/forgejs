## 1. Purpose

The Foundry registry protocol defines the HTTP API that the Forge toolchain uses to publish, resolve, and download packages. It is the wire contract between:

- **`forge publish`** — CLI command that uploads a package to the registry.
- **`forge install`** — CLI command that resolves and downloads dependencies.
- **Registry implementations** — the canonical registry at `registry.forgejs.com` and any self-hosted equivalents.

Any HTTP server that correctly implements this protocol is a valid Foundry registry. The Forge CLI discovers the registry URL from `forge.toml` or `foundry.toml` (defaulting to `https://registry.forgejs.com`) and communicates exclusively via this protocol.

---

## 2. Base URL and Hosting

**Canonical registry:** `https://registry.forgejs.com`

**Custom registry:** A project may specify an alternative registry URL in `forge.toml`:

```toml
[registry]
url = "https://registry.mycompany.com"
```

All paths in this spec are relative to the base URL. The base URL must use HTTPS. HTTP-only registries are rejected by the CLI.

**Path prefix:** Self-hosted registries may be mounted under a path prefix (e.g. `https://internal.mycompany.com/forge-registry/`). The base URL includes the prefix; all API paths are appended to it.

---

## 3. Authentication

Authentication uses **API keys** passed as a Bearer token in the `Authorization` header:

```text
Authorization: Bearer fry_sk_abc123def456...
```

**Key format:** `fry_sk_<alphanumeric>` (prefix `fry_sk_` followed by at least 32 alphanumeric characters). Keys are generated by the registry; the format is opaque to the CLI.

**Key scopes:**

| Scope | Allows |
|-------|--------|
| `publish:<author>` | Publish packages under the `<author>` namespace |
| `read` | Download packages (implicit for public packages; required for private) |
| `admin` | Claim namespaces, manage API keys |

**Anonymous access:** Public packages may be downloaded and searched without authentication. The `Authorization` header is optional for `GET /packages` and `GET /search`.

**Authentication errors:**

| Condition | Status | Error code |
|-----------|--------|------------|
| Malformed `Authorization` header | 400 | `invalid_auth_header` |
| Key does not exist | 401 | `invalid_api_key` |
| Key lacks required scope | 403 | `insufficient_scope` |
| Key has been revoked | 401 | `api_key_revoked` |

---

## 4. Versioning and Compatibility

The protocol is versioned via an API version header:

```text
X-Forge-Protocol: 1
```

The CLI sends this header on all requests. If the registry does not recognise the version, it returns 400 with error code `unsupported_protocol_version`.

**Current version:** `1`

## 5. Endpoints

### 5.1 POST /packages/{author}/{name}

Publish a new version of a package.

**Authentication:** Required. Key must have `publish:<author>` scope. If the `<author>` namespace is not owned by the key's account, returns 403.

**Path parameters:**

- `author` — the author component of the package name.
- `name` — the name component.

**Request body:** `multipart/form-data` with exactly two parts:

| Part name | Content-Type | Description |
|-----------|-------------|-------------|
| `manifest` | `text/plain;charset=utf-8` | The full text of the package's `foundry.toml`. |
| `tarball` | `application/x-tar` | The gzip-compressed tar archive of the package source (see spec 003 §5 for the tarball format). |

The version number is read from the `manifest` part. It must match the `<author>/<name>` in the path. If they differ, 400 is returned.

**Success Response — 201 Created:**

```json
{
  "name": "jeshua/date-utils",
  "version": "2.1.0",
  "integrity": "blake3:a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2",
  "url": "https://registry.forgejs.com/packages/jeshua/date-utils/2.1.0",
  "published_at": "2026-03-30T12:34:56Z"
}
```

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 400 | `invalid_manifest` | `foundry.toml` fails validation (see spec 003 §2.4) |
| 400 | `name_mismatch` | Package name in manifest does not match URL path |
| 400 | `invalid_tarball` | Tarball is not a valid gzip-compressed tar archive |
| 400 | `tarball_too_large` | Tarball exceeds 50 MB |
| 400 | `missing_src_directory` | Tarball does not contain a `src/` directory |
| 400 | `breaking_change_without_major_bump` | Static analysis detected a breaking API change without a major version increment (see §5.1.1) |
| 401 | `authentication_required` | No `Authorization` header |
| 401 | `invalid_api_key` | Key does not exist |
| 403 | `insufficient_scope` | Key lacks `publish:<author>` scope |
| 403 | `namespace_not_owned` | `<author>` namespace is not owned by this key's account |
| 409 | `version_exists` | This exact version already exists (versions are immutable) |
| 422 | `reserved_namespace` | `<author>` is `"forge"` (reserved) |

#### 5.1.1 Breaking Change Detection

The registry performs a static analysis comparison between the new version's exported type signatures and the previous latest version. If a breaking change is detected (removed export, narrowed parameter type, broadened return type) without a major version bump, the publish is rejected with error code `breaking_change_without_major_bump`.

This check is advisory in `0.x` versions (major = 0) — it emits a warning but does not block publish.

### 5.2 GET /packages/{author}/{name}/{version}

Download the source tarball for a specific version.

**Authentication:** Not required for public packages. Required for private packages (key with `read` scope).

**Path parameters:**

- `author` — author namespace.
- `name` — package name.
- `version` — exact version string (e.g. `2.1.0`).

**Success Response — 200 OK:**

- `Content-Type: application/x-tar`
- `Content-Encoding: gzip`
- `X-Forge-Integrity: blake3:<64-char hex>`
- `X-Forge-Version: 2.1.0`
- `X-Forge-Name: jeshua/date-utils`
- `Cache-Control: public, immutable, max-age=31536000`

Body: the raw gzip-compressed tar archive.

**Caching:** Versioned package downloads are immutable. The `Cache-Control: public, immutable` header enables aggressive CDN and client-side caching. The Forge CLI caches downloaded tarballs in `.forge/packages/<author>/<name>/<version>/` and verifies the BLAKE3 hash before using the cache.

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 404 | `package_not_found` | Package `author/name` does not exist |
| 404 | `version_not_found` | Package exists but this version does not |
| 410 | `version_yanked` | Version exists but has been yanked |

---

### 5.3 GET /packages/{author}/{name}

Retrieve package metadata including all available versions.

**Authentication:** Not required for public packages.

**Path parameters:**

- `author`, `name` — as above.

**Success Response — 200 OK:**

```json
{
  "name": "jeshua/date-utils",
  "description": "Lightweight date formatting and parsing utilities for Forge applications",
  "license": "MIT",
  "homepage": "https://github.com/jeshua/date-utils",
  "latest": "2.1.0",
  "versions": [
    {
      "version": "1.0.0",
      "published_at": "2025-06-15T08:00:00Z",
      "yanked": false
    },
    {
      "version": "2.0.0",
      "published_at": "2025-11-01T09:30:00Z",
      "yanked": false
    },
    {
      "version": "2.1.0",
      "published_at": "2026-03-30T12:34:56Z",
      "yanked": false
    }
  ],
  "keywords": ["date", "time", "formatting"]
}
```

**The `latest` field** reflects the highest non-yanked SemVer version. Pre-release versions (e.g. `1.0.0-pre-alpha`) are not returned as `latest` unless all versions are pre-release.

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 404 | `package_not_found` | Package does not exist |

### 5.4 POST /resolve

Resolve a set of package name/version pairs to a full dependency graph, including all transitive dependencies.

**Authentication:** Not required.

**Request body:** `application/json`

```json
{
  "packages": [
    { "name": "jeshua/date-utils", "version": "*" },
    { "name": "acme/ui-kit", "version": "3.0.0" }
  ]
}
```

**Version field:** Either an exact version string or `"*"` (resolve to latest stable).

**Success Response — 200 OK:**

```json
{
  "resolved": [
    {
      "name": "jeshua/date-utils",
      "version": "2.1.0",
      "integrity": "blake3:a1b2c3d4...",
      "url": "https://registry.forgejs.com/packages/jeshua/date-utils/2.1.0",
      "dependencies": [
        { "name": "jeshua/locale-data", "version": "1.0.4" }
      ]
    },
    {
      "name": "acme/ui-kit",
      "version": "3.0.0",
      "integrity": "blake3:d4e5f6a7...",
      "url": "https://registry.forgejs.com/packages/acme/ui-kit/3.0.0",
      "dependencies": [
        { "name": "jeshua/date-utils", "version": "2.1.0" }
      ]
    },
    {
      "name": "jeshua/locale-data",
      "version": "1.0.4",
      "integrity": "blake3:f1e2d3c4...",
      "url": "https://registry.forgejs.com/packages/jeshua/locale-data/1.0.4",
      "dependencies": []
    }
  ]
}
```

The `resolved` array contains **all** packages in the dependency graph (direct and transitive), deduplicated. Packages are sorted lexicographically by name then version.

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 400 | `invalid_request` | Malformed JSON or missing `packages` field |
| 400 | `invalid_package_name` | A package name does not match `author/name` format |
| 400 | `invalid_version` | A version string is not a valid SemVer or `"*"` |
| 404 | `package_not_found` | A requested package does not exist |
| 404 | `version_not_found` | A requested exact version does not exist |
| 409 | `dependency_conflict` | Two packages in the graph require different exact versions of the same transitive dependency |

**Conflict error body:**

```json
{
  "error": "dependency_conflict",
  "message": "Dependency conflict detected",
  "conflicts": [
    {
      "package": "jeshua/crypto",
      "required_by": [
        { "name": "jeshua/auth-helpers", "version": "1.0.0", "requires": "2.0.0" },
        { "name": "acme/ui-kit", "version": "3.0.0", "requires": "2.1.0" }
      ]
    }
  ]
}
```

### 5.5 GET /search

Search the package registry.

**Authentication:** Not required.

**Query parameters:**

| Parameter | Type | Required | Default | Description |
|-----------|------|----------|---------|-------------|
| `q` | string | **Yes** | — | Search query. Matched against name, description, keywords, and author. |
| `limit` | integer | No | `20` | Results per page. Maximum `100`. |
| `offset` | integer | No | `0` | Number of results to skip. |
| `author` | string | No | — | Filter by author namespace. |
| `sort` | string (enum) | No | `relevance` | Sort order. One of: `relevance`, `downloads`, `updated`. |

**Success Response — 200 OK:**

```json
{
  "total": 42,
  "limit": 20,
  "offset": 0,
  "results": [
    {
      "name": "jeshua/date-utils",
      "version": "2.1.0",
      "description": "Lightweight date formatting and parsing utilities for Forge applications",
      "license": "MIT",
      "keywords": ["date", "time", "formatting"],
      "downloads_last_30_days": 1240,
      "published_at": "2026-03-30T12:34:56Z"
    }
  ]
}
```

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 400 | `missing_query` | `q` parameter is absent or empty |
| 400 | `invalid_limit` | `limit` is not a positive integer or exceeds 100 |
| 400 | `invalid_offset` | `offset` is negative |

### 5.6 POST /namespaces/{author}/claim

Claim an author namespace. After claiming, only API keys associated with the claiming account may publish packages under this namespace.

**Authentication:** Required. Key must have `admin` scope.

**Path parameters:**

- `author` — the namespace to claim. Must match `^[a-z][a-z0-9-]*$`. Must not be `forge` (reserved).

**Request body:** `application/json`

```json
{
  "account_id": "acc_abc123"
}
```

**Success Response — 201 Created:**

```json
{
  "author": "jeshua",
  "claimed_by": "acc_abc123",
  "claimed_at": "2026-03-30T12:00:00Z"
}
```

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 400 | `invalid_namespace` | Author name fails format validation |
| 403 | `reserved_namespace` | Namespace is `forge` or otherwise reserved |
| 409 | `namespace_already_claimed` | Another account already owns this namespace |

### 5.8 DELETE /packages/{author}/{name}/{version}

Yank a specific version. Yanking marks a version as deprecated/dangerous without deleting it. Yanked versions:

- Are not returned as the `latest` version.
- Are not included in `POST /resolve` responses for `"*"` version constraints.
- Can still be downloaded by exact version if already pinned in `foundry.lock`.
- Are listed in `GET /packages/{author}/{name}` with `"yanked": true`.

**Authentication:** Required. Key must have `publish:<author>` scope.

**Request body:** `application/json`

```json
{
  "reason": "Security vulnerability CVE-2026-1234 — upgrade to 2.1.1"
}
```

**Success Response — 200 OK:**

```json
{
  "name": "jeshua/date-utils",
  "version": "2.1.0",
  "yanked": true,
  "yank_reason": "Security vulnerability CVE-2026-1234 — upgrade to 2.1.1",
  "yanked_at": "2026-03-30T14:00:00Z"
}
```

**Error Responses:**

| Status | Error code | Condition |
|--------|-----------|-----------|
| 400 | `missing_reason` | `reason` field absent or empty |
| 404 | `version_not_found` | Package/version does not exist |
| 409 | `already_yanked` | Version is already yanked |

**Unyanking:** Not supported. Once yanked, a version remains yanked. Publish a new patch version instead.

### 5.9 GET /health

Registry health check endpoint.

**Authentication:** Not required.

**Success Response — 200 OK:**

```json
{
  "status": "ok",
  "version": "1.0.0",
  "protocol": 1
}
```

**Degraded Response — 200 OK:**

```json
{
  "status": "degraded",
  "version": "1.0.0",
  "protocol": 1,
  "message": "Search index is rebuilding — search results may be incomplete"
}
```

The Forge CLI checks this endpoint before long operations (`forge publish`, `forge install`) and warns the user if the registry is degraded.

## 6. Error Response Format

All error responses use `application/json` with the following structure:

```json
{
  "error": "<error_code>",
  "message": "<human-readable description>"
}
```

For conflict errors (409 `dependency_conflict`), a `conflicts` field is added as described in §5.4.

For validation errors (400), a `fields` array may be included:

```json
{
  "error": "invalid_manifest",
  "message": "foundry.toml validation failed",
  "fields": [
    {
      "field": "package.version",
      "message": "must be a valid SemVer string",
      "value": "not-a-version"
    }
  ]
}
```

**HTTP status codes used:**

| Status | Meaning |
|--------|---------|
| 200 | Success |
| 201 | Created (publish success) |
| 400 | Bad request (validation error, malformed input) |
| 401 | Unauthenticated |
| 403 | Forbidden (authenticated but not authorised) |
| 404 | Not found |
| 405 | Method not allowed |
| 409 | Conflict (version already exists, dependency conflict, namespace taken) |
| 410 | Gone (yanked) |
| 422 | Unprocessable entity (semantic error, e.g. reserved namespace) |
| 429 | Too many requests (rate limited) |
| 500 | Registry internal error |
| 503 | Registry unavailable |

## 7. Rate Limits

Rate limits are applied per source IP for anonymous requests and per API key for authenticated requests.

| Client type | Limit | Window |
|-------------|-------|--------|
| Anonymous | 100 requests | 1 minute |
| Authenticated (any key) | 1000 requests | 1 minute |
| Publish (`POST /packages/**`) | 10 requests | 1 hour per namespace |

**Rate limit headers** are included on all responses:

```text
X-RateLimit-Limit: 1000
X-RateLimit-Remaining: 987
X-RateLimit-Reset: 1743340496
```

`X-RateLimit-Reset` is a Unix timestamp indicating when the window resets.

**Rate limit exceeded — 429 Too Many Requests:**

```json
{
  "error": "rate_limited",
  "message": "Rate limit exceeded. Try again after 2026-03-30T13:00:00Z",
  "retry_after": "2026-03-30T13:00:00Z"
}
```

The `Retry-After` HTTP header is also set with the same timestamp.

The Forge CLI automatically retries rate-limited requests with exponential backoff, up to 3 retries.

## 8. Integrity Verification Protocol

The Forge CLI enforces BLAKE3 integrity verification for all downloaded packages. This section defines the normative verification steps.

**After `GET /packages/{author}/{name}/{version}`:**

1. Read the expected hash from the `X-Forge-Integrity` response header.
2. Compute the BLAKE3 hash of the entire response body (the raw bytes of the gzip-compressed tarball).
3. Compare the computed hash to the expected hash using a constant-time comparison function.
4. If they match: extract the tarball to `.forge/packages/<author>/<name>/<version>/` and proceed.
5. If they do not match:
   a. Delete any partially extracted files.
   b. Remove the cached tarball if any.
   c. Emit the error:

   ```text
   error: integrity verification failed for jeshua/date-utils@2.1.0
     expected: blake3:a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6...
     got:      blake3:f9e8d7c6b5a4f3e2d1c0b9a8f7e6d5c4...
   ```

   d. Abort the `forge install` operation entirely.

**Against `foundry.lock`:**

When `forge install --frozen` is run (CI mode), the expected hash is read from `foundry.lock`, not from the response header. Both must agree: if the `X-Forge-Integrity` header differs from `foundry.lock`, the download is rejected regardless of whether the computed hash matches the header.

**Cache reuse:**

The CLI caches tarballs in `.forge/packages/`. Before re-downloading, it checks whether the cached tarball's BLAKE3 hash matches `foundry.lock`. If it matches, the network download is skipped. This check must use the same constant-time comparison as the download verification.

## 9. Namespace Ownership

- A namespace (`author`) is **claimed** by the first account to call `POST /namespaces/{author}/claim`.
- Before a namespace is claimed, anyone may publish to it. After claiming, only the owning account may publish.
- The canonical registry enforces namespace ownership globally. A self-hosted registry may implement more permissive policies.
- Namespace ownership cannot be transferred via the API in this version. Contact registry admins for ownership disputes.

---

## 10. Package Immutability and Yanking

- Published versions are **immutable**. The registry never modifies the content of a published version.
- Yanking (§5.8) deprecates a version without removing it. This preserves reproducibility for projects already pinned to the yanked version.
- Hard deletion is not supported via the API. Registry operators may perform physical deletion in exceptional circumstances (e.g. DMCA, accidental secret exposure), but this breaks reproducibility for consumers.

---

## 11. Self-Hosted Registry

Any HTTP server implementing the endpoints in §5 is a valid Foundry registry. Self-hosted registries:

- Must implement at minimum: `POST /packages`, `GET /packages/{author}/{name}/{version}`, `GET /packages/{author}/{name}`, `POST /resolve`, `GET /health`.
- May implement a subset of optional endpoints (`GET /search`, `POST /namespaces/{author}/claim`, etc.).
- Must return the `X-Forge-Protocol: 1` header on all responses.
- Must enforce HTTPS.
- Must return `X-Forge-Integrity` headers on tarball responses.

**Registry configuration in `forge.toml`:**

```toml
[registry]
url = "https://registry.mycompany.com"
```

**Per-package registry override** (to mix packages from the canonical registry and a private registry) is not supported in this version. All packages in a project resolve from the single configured registry.
