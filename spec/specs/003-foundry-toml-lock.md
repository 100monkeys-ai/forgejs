## 1. Purpose

A Forge **package** is a reusable library published to the Foundry registry. Every package has two manifest files:

- **`foundry.toml`** — the package's own manifest. Declares identity, dependencies, and metadata. Published alongside source code to the registry.
- **`foundry.lock`** — the deterministic resolution snapshot. Generated by the Forge toolchain. Must be committed to version control. Never edited by hand.

These are distinct from `forge.toml`, which is the **application** manifest (see spec 002). An application's `forge.toml` declares its dependencies; a package's `foundry.toml` declares the package's own identity and its transitive dependencies.

A directory that contains a `foundry.toml` is a **Foundry package**. A directory that contains a `forge.toml` is a **Forge application**. A directory may not contain both.

## 2. `foundry.toml` — Package Manifest

### 2.1 Schema Reference

#### `[package]`

The `[package]` table is **required**. It defines the published identity of the package.

| Field | Type | Required | Default | Description |
|-------|------|----------|---------|-------------|
| `name` | string | **Yes** | — | Fully-qualified package name in `author/name` format. |
| `version` | string | **Yes** | — | SemVer 2.0.0 version string. No ranges. |
| `description` | string | No | `""` | One-line human description. Maximum 256 characters. |
| `license` | string | No | `""` | SPDX license identifier. |
| `keywords` | array of string | No | `[]` | Search keywords. Maximum 10 entries, 32 chars each. |
| `homepage` | string | No | `""` | Project homepage URL. |

#### `[dependencies]`

A TOML table mapping `author/name` to a version constraint. Identical syntax to `forge.toml` dependencies (exact version or `"*"`). It is **optional**.

#### `[dev-dependencies]`

Same format as `[dependencies]`. Packages listed here are used only during the package's own development and testing. They are **not** included in the resolved dependency graph when this package is used as a dependency of another package.

#### `[peer-dependencies]`

Same format as `[dependencies]`. Peer dependencies declare compatibility requirements without pulling in a copy. When a consumer installs this package, the peer dependency must already be present in the consumer's dependency graph at a compatible version. If it is absent, the Forge resolver emits a warning (not an error).

**Peer dependency version matching:** Because Forge uses exact versions only, a peer dependency version of `"1.2.3"` means the consumer must have exactly `"1.2.3"` of that package. A peer dependency version of `"*"` means any version is acceptable.

---

### 2.2 Package Naming Rules

**Format:** `author/name`

Both the author component and the name component independently obey:

```text
^[a-z][a-z0-9-]*$
```

- Must begin with a lowercase ASCII letter.
- Subsequent characters: lowercase ASCII letters, ASCII digits, or hyphens.
- No uppercase letters, underscores, dots, or other punctuation.

**Length constraints:**

- `author`: minimum 1, maximum 39 characters.
- `name`: minimum 1, maximum 64 characters.
- Total (`author` + `/` + `name`): maximum 100 characters.

**Reserved namespace:** `forge` is a reserved author name. Only the Forge toolchain may publish packages under the `forge/` namespace. Attempting to publish with `author = "forge"` is rejected by the registry with HTTP 403.

**`forge:*` specials:** The specifier prefix `forge:` (used in imports, e.g. `import { db } from 'forge:db'`) refers to FSL packages built into the Forge toolchain. These do not have `foundry.toml` manifests and cannot be published via the Foundry protocol.

**Immutability:** Once a package at a given `author/name@version` is published, it cannot be modified. Yanking (marking a version as deprecated/dangerous) is supported by the registry API but does not delete the package.

### 2.3 Complete Example

```toml
[package]
name = "jeshua/date-utils"
version = "2.1.0"
description = "Lightweight date formatting and parsing utilities for Forge applications"
license = "MIT"
keywords = ["date", "time", "formatting", "parsing"]
homepage = "https://github.com/jeshua/date-utils"

[dependencies]
"jeshua/locale-data" = "1.0.4"

[dev-dependencies]
"forge/test-utils" = "*"

[peer-dependencies]
"acme/i18n-core" = "*"
```

### 2.4 Validation Rules

| Rule | Error |
|------|-------|
| `[package]` table missing | `foundry.toml: missing required table [package]` |
| `package.name` missing | `foundry.toml: package.name is required` |
| `package.name` not in `author/name` format | `foundry.toml: package.name must be in author/name format` |
| `author` part invalid characters | `foundry.toml: author component must match ^[a-z][a-z0-9-]*$` |
| `name` part invalid characters | `foundry.toml: name component must match ^[a-z][a-z0-9-]*$` |
| `author` is `"forge"` | `foundry.toml: 'forge' is a reserved author namespace` |
| Total name length > 100 | `foundry.toml: package name exceeds 100 character limit` |
| `package.version` missing | `foundry.toml: package.version is required` |
| `package.version` not SemVer | `foundry.toml: package.version must be a valid SemVer string` |
| `package.version` contains build metadata | `foundry.toml: package.version must not contain build metadata (+)` |
| More than 10 keywords | `foundry.toml: keywords may not exceed 10 entries` |
| Keyword > 32 chars | `foundry.toml: keyword '<word>' exceeds 32 character limit` |
| Dependency uses range syntax | `foundry.toml: dependency '<name>' uses a version range; only exact versions or "*" are supported` |
| Self-dependency | `foundry.toml: package may not depend on itself` |

## 3. `foundry.lock` — Lockfile

### 3.1 Format

`foundry.lock` is a TOML file generated by the Forge toolchain. It records the complete, deterministic resolution of all direct and transitive dependencies. It must be committed to version control alongside `foundry.toml` (or `forge.toml` for applications).

The file begins with a two-line header:

```toml
# foundry.lock — generated by forge, do not edit manually
# Forge version: 0.1.0-pre-alpha
```

The Forge version in the header is the version of the toolchain that last regenerated the lockfile. It is informational only and does not gate compatibility.

After the header, the file contains one `[[package]]` array-of-tables entry per resolved package (direct and transitive), sorted lexicographically by `name` then `version`.

---

### 3.2 Entry Fields

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `name` | string | **Yes** | Fully-qualified package name (`author/name`). |
| `version` | string | **Yes** | Exact resolved version. |
| `integrity` | string | **Yes** | BLAKE3 hash of the package tarball. Format: `blake3:<64-char lowercase hex>`. |
| `url` | string | **Yes** | Full URL to the package tarball on the registry. |
| `dependencies` | array of inline tables | No | Direct dependencies of this package. Each entry has `name` and `version` fields. Omitted if the package has no dependencies. |

All fields are written as strings. No integer or boolean values appear in lockfile entries.

---

### 3.3 Example Lockfile

```toml
# foundry.lock — generated by forge, do not edit manually
# Forge version: 0.1.0-pre-alpha

[[package]]
name = "acme/ui-kit"
version = "3.0.0"
integrity = "blake3:d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2c3d4e5"
url = "https://registry.forgejs.com/packages/acme/ui-kit/3.0.0.tar.gz"
dependencies = [
  { name = "jeshua/date-utils", version = "2.1.0" },
]

[[package]]
name = "jeshua/date-utils"
version = "2.1.0"
integrity = "blake3:a1b2c3d4e5f6a7b8c9d0e1f2a3b4c5d6e7f8a9b0c1d2e3f4a5b6c7d8e9f0a1b2"
url = "https://registry.forgejs.com/packages/jeshua/date-utils/2.1.0.tar.gz"
dependencies = [
  { name = "jeshua/locale-data", version = "1.0.4" },
]

[[package]]
name = "jeshua/locale-data"
version = "1.0.4"
integrity = "blake3:f1e2d3c4b5a6f7e8d9c0b1a2f3e4d5c6b7a8f9e0d1c2b3a4f5e6d7c8b9a0f1e2"
url = "https://registry.forgejs.com/packages/jeshua/locale-data/1.0.4.tar.gz"
```

---

### 3.4 Lockfile Lifecycle

**Generation:** `forge install` resolves all dependencies and writes `foundry.lock`. If a `foundry.lock` already exists, it is used as the basis for resolution — only packages not already in the lockfile are resolved from the registry.

**Integrity check:** `forge install --frozen` (used in CI) reads the existing `foundry.lock` exactly. It does not resolve from the registry. It downloads each package and verifies the `integrity` hash. If any hash mismatches or the lockfile is absent, the command fails.

**Update:** `forge update <author/name>` re-resolves a single package (fetching the latest version if pinned as `"*"`) and updates the corresponding `[[package]]` entry and all affected transitive entries.

**Out-of-date detection:** If `forge.toml` (or `foundry.toml`) declares a dependency that is absent from `foundry.lock`, or a lockfile entry references a package not in the manifest, the CLI emits a warning and prompts to run `forge install`.

**Conflict on merge:** If two branches modify `foundry.lock` concurrently, the merge will produce a conflict. The correct resolution is always to run `forge install` after resolving the `foundry.toml` conflict — never manually merge `foundry.lock` entries.

## 4. Resolution Algorithm

Forge uses **exact-version pinning** with a flat dependency graph. There are no version ranges; every dependency must specify either an exact version or `"*"` (resolve to latest).

### Algorithm Steps

1. **Collect direct dependencies** from `forge.toml` (for applications) or `foundry.toml` (for packages). Include both `[dependencies]` and `[peer-dependencies]`; exclude `[dev-dependencies]` unless resolving for development.

2. **Consult the lockfile.** For each direct dependency:
   - If the package is already in `foundry.lock` at the declared version (or any version if declared as `"*"`), use the locked version. Do not contact the registry.
   - If the package is not in `foundry.lock`, proceed to step 3.

3. **Resolve from registry.** For packages not in the lockfile:
   - If version is `"*"`: call `GET /packages/{author}/{name}` and use the `latest` field.
   - If version is exact: the locked version is the declared version.

4. **Fetch transitive dependencies.** For each resolved package, fetch its `foundry.toml` from the registry (bundled in the tarball) and collect its `[dependencies]`. Apply the same resolution logic recursively.

5. **Detect conflicts.** If two packages in the resolved graph require different exact versions of the same package, resolution fails with:

   ```text
   error: dependency conflict
     jeshua/auth-helpers@1.0.0 requires jeshua/crypto@2.0.0
     acme/ui-kit@3.0.0 requires jeshua/crypto@2.1.0
     These versions cannot coexist. Upgrade or downgrade one of the depending packages.
   ```

6. **Write lockfile.** Emit a new `foundry.lock` with all resolved packages sorted lexicographically.

### Conflict Resolution Policy

Conflicts are **hard errors**. There is no automatic conflict resolution heuristic. The developer must resolve the conflict by:

- Upgrading or downgrading one of the conflicting packages in their manifest.
- Contacting the package authors to align on a shared transitive dependency version.

This strict policy ensures deterministic, auditable builds at the cost of occasionally requiring manual version alignment.

## 5. Integrity Format

**Format:** `blake3:<64-char lowercase hexadecimal string>`

The BLAKE3 hash is computed over the **package source tarball**. The tarball is defined as:

1. A gzip-compressed tar archive (`.tar.gz`).
2. The archive contains exactly two entries:
   - `foundry.toml` — the package manifest.
   - `src/` — a directory tree containing all source files.
3. File entries within `src/` are sorted lexicographically by path (for determinism).
4. Tar entry metadata (timestamps, ownership) is normalised to zero before hashing to ensure reproducible hashes regardless of build environment.

**Verification:** After downloading a package tarball, the Forge toolchain:

1. Reads the expected hash from `foundry.lock`.
2. Computes the BLAKE3 hash of the downloaded bytes.
3. Compares the two hashes using a constant-time comparison.
4. If they differ: rejects the tarball, deletes any partially extracted files, and emits:

   ```text
   error: integrity verification failed for jeshua/date-utils@2.1.0
     expected: blake3:a1b2c3d4...
     got:      blake3:f9e8d7c6...
   ```

5. The cached copy (in `.forge/packages/`) is also invalidated and re-downloaded.

**Hash algorithm rationale:** BLAKE3 is chosen for its cryptographic strength, resistance to length-extension attacks, and performance characteristics suitable for large tarballs.

## 6. Relationship Between `forge.toml` and `foundry.toml`

| Property | `forge.toml` | `foundry.toml` |
|----------|-------------|----------------|
| Belongs to | Application | Package/library |
| Published to registry | No | Yes (inside tarball) |
| Contains `[[target]]` | Yes | No |
| Contains `[build]` / `[dev]` / `[server]` | Yes | No |
| Contains `[peer-dependencies]` | No | Yes |
| Name format | Simple (`my-app`) | `author/name` |
| Version | Informational | Published; immutable per version |
| Lockfile | `foundry.lock` | `foundry.lock` |

Both file types share the same `[dependencies]` / `[dev-dependencies]` schema and the same version constraint syntax. Both generate or consume the same `foundry.lock` format.

A directory containing both `forge.toml` and `foundry.toml` is invalid. The CLI emits an error:

```text
error: directory contains both forge.toml and foundry.toml
  A Forge project cannot be both an application and a package.
  Remove one of the manifest files.
```
